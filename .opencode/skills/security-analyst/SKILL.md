# Security Analyst — Wazuh Alert Investigation Skill

## Purpose

Analyze security alert digest reports (Markdown format) from a Wazuh AI-triaged notification system. These reports are generated by an n8n workflow that:

1. Receives Wazuh alerts (level 10+) via syslog integration
2. Batches them hourly
3. Sends them to Gemini AI for initial triage and analysis
4. Delivers the digest as a Markdown email attachment

Your job is to go deeper than the AI triage — correlate alerts, investigate on the live system, assess real risk in context, run proactive security audits, cross-reference known vulnerabilities, and produce actionable remediation.

## When to Use This Skill

- The user provides a security alert report (markdown file) from the Wazuh notification system
- The user asks to investigate a security incident or suspicious activity
- The user asks to assess vulnerabilities or propose fixes for the environment
- The user wants to review Wazuh alert patterns or tune detection rules
- The user asks about the security posture of the environment
- The user asks to run a proactive security scan or vulnerability assessment

## Security Principles

1. **NEVER read or display the contents of `.env` or `terraform.tfvars`** — they contain secrets.
2. **NEVER commit investigation reports, false positive catalogs, or vulnerability scan results to git.** These files reveal the system's weaknesses and must remain local-only.
3. **Always check the system** before concluding an alert is benign. Do not guess.
4. **Correlate before investigating** — grouped alerts save time and give better context.
5. **Be specific in remediation** — "harden SSH" is useless. "Add `PermitRootLogin no` to `/etc/ssh/sshd_config` line 32" is useful.
6. **Explain risk** — for every remediation, state what happens if the user does NOT apply it.
7. **Preserve existing hardening** — discover the current security posture before proposing changes. Do not weaken what is already in place.
8. **Prioritize by exploitability** — a critical CVE on an internet-facing service matters more than a misconfiguration on an internal-only tool.

## Environment Discovery

Understand the architecture before making any assessment. Do NOT assume the number of hosts, the topology, or the scale.

### Infrastructure Discovery

Before starting any investigation, you MUST discover the host environment. Do NOT assume the hardware, OS, or distribution. Run these commands to gather context:

```bash
# Host hardware and OS
hostnamectl
uname -a
lsb_release -a 2>/dev/null || cat /etc/os-release
nproc && free -h && df -h /

# Container runtime
docker --version
docker compose version
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Image}}" | sort

# Wazuh agent identity (discover the manager container name from docker ps first)
WAZUH_MGR=$(docker ps --format '{{.Names}}' | grep -i 'wazuh.*manager' | head -1)
[ -n "$WAZUH_MGR" ] && docker exec "$WAZUH_MGR" /var/ossec/bin/agent_control -l 2>/dev/null || true

# Network posture
ss -tulnp | head -40

# Firewall status
sudo ufw status verbose 2>/dev/null || sudo iptables -L -n --line-numbers 2>/dev/null | head -30
```

Use the outputs to build your understanding of:
- **Host**: Hardware model, OS and version, kernel, CPU/RAM/disk
- **Containers**: How many are running, what images, what orchestrator
- **Wazuh agent**: Agent name and ID for host alerts
- **Network exposure**: What ports are listening and on which interfaces
- **Firewall**: What rules are active, what is the default policy

### Known Infrastructure Patterns

These are architectural patterns to be aware of — discover the specifics during Infrastructure Discovery:

- **Ingress**: Traffic enters through a tunnel proxy container. Discover which container and what ports are exposed by checking `docker ps` and `ss -tulnp`.
- **Access control**: Public-facing services may be behind an identity-aware proxy or zero-trust policy. Discover the access model by examining Terraform files (if present) or the tunnel configuration.
- **Container hardening**: Containers may have security options applied. Discover them via `docker inspect --format '{{.Name}} CapDrop={{.HostConfig.CapDrop}} CapAdd={{.HostConfig.CapAdd}} SecurityOpt={{.HostConfig.SecurityOpt}} ReadOnly={{.HostConfig.ReadonlyRootfs}}' $(docker ps -q)`.

### Container Inventory

Do NOT rely on a hardcoded container list. Discover the running containers during Infrastructure Discovery using `docker ps`. Cross-reference with the `docker-compose.yml` in the project root to understand each container's purpose and expected behavior.

### Project File Locations

Do NOT assume absolute paths. Discover the project root by looking for the `docker-compose.yml` file in the current working directory or the user's project directory. Key files to locate:

```bash
# Find the project root (contains docker-compose.yml)
# Then explore the structure from there:
ls -la           # Look for docker-compose.yml, .env, and config directories
ls -la files/ 2>/dev/null || ls -la config/ 2>/dev/null  # Service configuration directories
ls -la terraform/ 2>/dev/null   # Infrastructure as code (if present)
```

Expect to find (names may vary):
- `docker-compose.yml` — Container definitions
- `.env` — Secrets and environment variables (**NEVER read or display**)
- A service configuration directory (e.g., `files/`, `config/`) — Service-specific configs
- A Terraform/OpenTofu directory (if IaC is used) — DNS, tunnel, access policies

## Local Data Store

All security-sensitive data produced by this skill is stored locally and NEVER committed to version control. The base path for local security data is `.local/security/` within the project root.

### False Positive Catalog

**File**: `.local/security/false-positives.yml`

This catalog is a machine-readable YAML file that accumulates known false positives over time. The skill reads it at the start of every investigation and updates it when new false positives are confirmed. This keeps the skill itself free of environment-specific data that could be exploited if the repository were compromised.

**Structure:**

```yaml
# .local/security/false-positives.yml
# Auto-maintained by the security-analyst skill. NEVER commit to git.
version: 1
last_updated: "YYYY-MM-DDTHH:MM:SSZ"
entries:
  - id: fp-001
    rule_id: <wazuh_rule_id>
    pattern: "<substring or regex matching the alert description or log line>"
    reason: "<why this is a false positive in this environment>"
    wazuh_suppression_rule: <custom_rule_id or null>
    added: "YYYY-MM-DD"
    last_seen: "YYYY-MM-DD"
    occurrences: 1
  # More entries added automatically during investigations
```

**Lifecycle:**
1. At investigation start, read the catalog (if it exists) and use it to pre-classify known FPs
2. When a new false positive is confirmed, append an entry with `id`, `rule_id`, `pattern`, `reason`, date, and optionally a `wazuh_suppression_rule` ID if one was created
3. When a known FP is seen again, increment `occurrences` and update `last_seen`
4. If the catalog file does not exist, create it with the `version: 1` header and an empty `entries: []` list

## Wazuh Knowledge Base

### Alert Severity Levels

| Level | Meaning | Action |
|-------|---------|--------|
| 0-4 | Low / informational | Ignored |
| 5-9 | Medium / notable | Logged only |
| 10-12 | High / suspicious | Forwarded to automation workflow for AI triage, included in digest |
| 13-15 | Critical / active threat | Forwarded to automation workflow for AI triage, included in digest |

Only alerts at level 10 and above reach the digest reports you will analyze.

### Rule Groups

- **syscheck** — File integrity monitoring (FIM). Detects file creation, modification, deletion, permission changes.
- **rootcheck** — Rootkit and malware detection.
- **syslog** — General system log analysis.
- **authentication** — Login attempts, sudo usage, SSH activity.
- **web** — HTTP/HTTPS access log analysis.
- **docker** — Container runtime events.
- **firewall** — iptables and network filtering events.
- **vulnerability-detector** — CVE scanning results.
- **sca** — Security Configuration Assessment (CIS benchmarks).

### Decoder Context

The decoder name in each alert tells you the log source:

- `sshd` — SSH daemon logs
- `sudo` — Privilege escalation via sudo
- `systemd` — Service manager events
- `pam` — Pluggable Authentication Modules
- `dpkg` — Package installation/removal
- `docker` — Docker daemon events
- `iptables` — Firewall rule matches
- `web-accesslog` — HTTP access logs (Nginx, etc.)

## Investigation Workflow

Follow these phases in order. Do NOT skip phases. Do NOT speculate without checking the system.

### Phase 0 — Proactive Attack Surface Audit

Run this audit at the START of every investigation, before even reading the digest. This catches problems that Wazuh alerts may not cover.

Launch a `general` subagent with the Task tool to perform these checks in parallel with Phase 1:

#### 0a. Network Exposure Scan

```bash
# Find all processes listening on 0.0.0.0 or :: (accepting connections from any interface)
ss -tulnp | grep -E '0\.0\.0\.0|::' | grep LISTEN

# Check Docker port mappings (host-bound ports that bypass firewall)
docker ps --format '{{.Names}}: {{.Ports}}' | grep '0.0.0.0'

# Check for Docker-published ports vs internal-only services
docker inspect --format '{{.Name}} {{.HostConfig.PortBindings}}' $(docker ps -q) | grep -v 'map\[\]'
```

Flag any service bound to `0.0.0.0` that should be `127.0.0.1` or internal Docker network only.

#### 0b. Container Hardening Audit

```bash
# Check every container for security posture
for cid in $(docker ps -q); do
  docker inspect --format '{{.Name}} | CapDrop={{.HostConfig.CapDrop}} | CapAdd={{.HostConfig.CapAdd}} | SecurityOpt={{.HostConfig.SecurityOpt}} | ReadOnly={{.HostConfig.ReadonlyRootfs}} | User={{.Config.User}} | Privileged={{.HostConfig.Privileged}}' "$cid"
done
```

Flag containers missing `no-new-privileges`, missing `cap_drop: [ALL]`, running as root without justification, or running privileged.

#### 0c. Authentication Exposure Check

```bash
# Look for containers with authentication disabled via environment variables
docker inspect --format '{{.Name}} {{.Config.Env}}' $(docker ps -q) | grep -iE 'auth.*false|no.auth|disable.*auth|anonymous'

# Check for services without password protection
docker inspect --format '{{.Name}} {{.Config.Env}}' $(docker ps -q) | grep -iE 'password|secret|token|key' | head -20
```

Flag any service with authentication explicitly disabled.

#### 0d. Firewall and Docker Bypass Assessment

```bash
# UFW status
sudo ufw status verbose 2>/dev/null

# Docker iptables chains (these bypass UFW)
sudo iptables -L DOCKER-USER -n -v 2>/dev/null
sudo iptables -L DOCKER -n -v 2>/dev/null | head -20
```

Document the firewall state and note that Docker-published ports bypass UFW's INPUT chain.

#### 0e. Host Patch Status

```bash
# Check for pending security updates
apt list --upgradable 2>/dev/null | head -30

# Check unattended-upgrades status
systemctl is-active unattended-upgrades 2>/dev/null
cat /etc/apt/apt.conf.d/20auto-upgrades 2>/dev/null

# Check kernel version vs available
uname -r
apt list --installed 2>/dev/null | grep linux-image | tail -3
```

Flag if security updates are pending or if unattended-upgrades is not active.

The subagent must return a structured attack surface report with findings categorized as CRITICAL / HIGH / MEDIUM / LOW.

### Phase 1 — Parse & Correlate (single general agent)

Launch ONE `general` subagent with the Task tool to read the digest report file and perform:

**Step 1: Parse the Report** — For each alert, extract:
- **Rule ID** and **rule level**
- **Description** (what Wazuh detected)
- **Agent** (check the Wazuh agent identity discovered during Infrastructure Discovery)
- **Timestamp** (check for time clustering — multiple alerts in seconds = possible attack chain)
- **Gemini's analysis** (the AI triage included in the report)
- **Raw log data** (the actual log line that triggered the alert)

**Step 2: Cross-reference the False Positive Catalog** — Read `.local/security/false-positives.yml` (if it exists). For each alert, check if `rule_id` + `pattern` matches a known FP. Mark matches as "Known FP" but still verify they haven't changed character (e.g., a previously benign rule now firing from a different source).

**Step 3: Correlate Alerts** — Look for patterns:
- **Time correlation**: Multiple alerts within the same minute may be a single event chain
- **Source correlation**: Same source IP or user across multiple alerts = coordinated activity
- **Kill chain progression**: Do the alerts map to sequential MITRE ATT&CK stages?
- **Repeated alerts**: Same rule firing many times = either ongoing attack or noisy false positive

**Step 4: Threat Assessment** — For each alert or alert group:
- MITRE ATT&CK classification (Tactic + Technique)
- Gemini triage validation (agree/disagree with AI, and why)
- Blast radius (what could an attacker reach from the compromised component?)
- Likelihood assessment (plausibly malicious given the environment context?)

**Step 5: Check Historical Context** — The orchestrator (you) must query the n8n data table for previous alerts BEFORE dispatching this subagent, then include the results in the subagent's prompt. Subagents do not have access to MCP tools.

**As the orchestrator, use the `n8n-mcp_n8n_manage_datatable` tool:**

```
Tool: n8n-mcp_n8n_manage_datatable
Parameters:
  action: "getRows"
  tableId: <discover the table ID by calling n8n-mcp_n8n_manage_datatable with action "listTables" first>
  limit: 100
  sortBy: "received_at:desc"
```

Look for a table that stores Wazuh alert data (typically named something like `wazuh_alerts`). Extract columns related to rule ID, severity level, description, agent name, and timestamp from recent rows. Summarize:
- How many times each rule ID has fired in the last 7/30 days
- Whether frequency is increasing, stable, or decreasing
- Any rule IDs from the current digest that have never been seen before (novel alerts)

Pass this historical summary to the Phase 1 subagent as context. If the n8n MCP tool is unavailable or the data table does not exist, skip historical context and note it in the report.

The agent must return:
- A structured list of all alerts with extracted fields and FP catalog matches
- Correlation groups (which alerts belong together and why)
- Historical context (previous occurrences from the data provided by the orchestrator)
- Initial MITRE ATT&CK classification for each group
- A prioritized list of investigation tasks per group (specific commands to run)

### Phase 2 — Live Investigation (parallel subagents per finding group)

For each correlated alert group from Phase 1, launch a SEPARATE `general` subagent in PARALLEL using the Task tool. Each subagent receives:
- The alert details and correlation context for its assigned group
- The specific investigation commands to run from Phase 1's output
- The host/container environment context discovered during Infrastructure Discovery
- Instructions to determine a verdict and propose remediation

#### Investigation Commands by Alert Type

**Host-level checks:**
```bash
# Recent authentication events
journalctl -u ssh --since "1 hour ago"
# Sudo usage
journalctl _COMM=sudo --since "1 hour ago"
# Currently logged-in users
who
last -10
# Running processes (look for anomalies)
ps auxf
# Network connections
ss -tulnp
# Cron jobs (persistence mechanism)
crontab -l
ls -la /etc/cron.d/
# Recently modified files
find /etc -mmin -60 -type f 2>/dev/null
# Check for unauthorized SSH keys
cat /home/*/.ssh/authorized_keys 2>/dev/null
```

**Docker-level checks:**
```bash
# Container status
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
# Specific container logs (replace CONTAINER)
docker logs --since 1h CONTAINER
# Check if any container has unexpected capabilities
docker inspect --format '{{.Name}} {{.HostConfig.CapAdd}}' $(docker ps -q)
# Check for containers running as root
docker inspect --format '{{.Name}} User={{.Config.User}}' $(docker ps -q)
```

**File integrity checks (when syscheck alerts fire):**
```bash
# Check the specific file mentioned in the alert
stat /path/to/file
md5sum /path/to/file
ls -la /path/to/file
git log --oneline -5 -- /path/to/file  # if tracked in git
```

**Network checks (when firewall or web alerts fire):**
```bash
# Check iptables rules
sudo iptables -L -n -v
# Check active connections to/from suspicious IPs
ss -tnp | grep SUSPICIOUS_IP
# DNS lookups for suspicious domains
dig SUSPICIOUS_DOMAIN
# Check tunnel/proxy container logs
docker logs --since 1h TUNNEL_CONTAINER
```

#### Compensating Controls Assessment

Each subagent MUST document the compensating controls that affect its findings. For each finding, note which of these mitigations apply:

- **Network isolation**: Is the affected service internal-only (no tunnel/proxy route) or public-facing?
- **Identity-aware proxy**: Is the service behind a zero-trust access policy (email OTP, etc.)?
- **Read-only filesystem**: Is the container read-only (`read_only: true`)?
- **Capability restrictions**: Does the container have `cap_drop: [ALL]`?
- **Non-root user**: Does the service run as a non-root user inside the container?
- **Secret isolation**: Are credentials injected via env vars and not baked into images?

This context is critical for accurate risk rating. A vulnerability behind three layers of access control is lower priority than one exposed directly.

Each subagent must return:
- Commands executed and their output (evidence)
- Compensating controls in effect for this finding
- Verdict: True Positive (Critical/Notable) or False Positive (Expected/Noisy) or Inconclusive
- MITRE ATT&CK classification (confirmed or revised)
- Proposed remediation with ready-to-execute commands or config changes

### Phase 3 — Vulnerability Assessment

After Phase 2 completes, launch a `general` subagent to perform a CVE/vulnerability cross-reference. This phase runs independently and its results are merged into the final report.

#### 3a. Container Image Scanning with Trivy

Trivy is run via Docker — no host installation required.

**CRITICAL: Trivy Supply Chain Safety**

Trivy itself was the target of a major supply chain attack (CVE-2026-33634, CVSS 9.4, CISA KEV). On March 19, 2026, attackers published a malicious Trivy v0.69.4 release that stole credentials. The attack also compromised `trivy-action` and `setup-trivy` GitHub Actions tags.

**Before running ANY Trivy scan, you MUST follow this safety protocol:**

1. **NEVER use `:latest` tag** — always pin to a specific known-safe version
2. **Verify the version** is not in the compromised set (v0.69.4 is the known malicious version)
3. **Check for new advisories** before pulling — the Trivy project may have been compromised again

```bash
# === TRIVY SAFETY PROTOCOL ===

# Step 1: Define the pinned safe version
# Known safe: v0.69.2, v0.69.3. Check https://github.com/aquasecurity/trivy/releases
# for the latest release and cross-reference against
# https://github.com/aquasecurity/trivy/security/advisories for any new incidents.
TRIVY_VERSION="0.69.3"

# Step 2: Check GitHub security advisories for new Trivy compromises
# Use the WebFetch tool to fetch:
#   https://github.com/aquasecurity/trivy/security/advisories
# If any NEW advisory exists beyond CVE-2026-33634 and CVE-2024-35192,
# STOP and alert the user before proceeding.

# Step 3: Pull the pinned version
docker pull "aquasec/trivy:${TRIVY_VERSION}" 2>/dev/null

# Step 4: Verify the image is from the official repository
# Check the image digest matches what's published on GitHub releases
docker inspect --format='{{index .RepoDigests 0}}' "aquasec/trivy:${TRIVY_VERSION}" 2>/dev/null

# Step 5: Quick self-test — the binary should report its own version
docker run --rm "aquasec/trivy:${TRIVY_VERSION}" version
# Verify the output matches TRIVY_VERSION. If it doesn't, DO NOT PROCEED.
```

**Compromised version reference (do NOT use):**
- `aquasec/trivy:0.69.4` — malicious, credential-stealing (CVE-2026-33634)
- Any `trivy-action` tag < 0.35.0 — force-pushed to malware
- Any `setup-trivy` tag < 0.2.6 — replaced with malicious commits

**After verifying safety, run the scans:**

```bash
TRIVY_VERSION="0.69.3"  # Must match the verified version above

# Scan all running container images
for img in $(docker ps --format '{{.Image}}' | sort -u); do
  echo "=== Scanning: $img ==="
  docker run --rm \
    -v /var/run/docker.sock:/var/run/docker.sock \
    -v trivy-cache:/root/.cache/ \
    "aquasec/trivy:${TRIVY_VERSION}" image \
    --severity HIGH,CRITICAL \
    --format json \
    --quiet \
    "$img" 2>/dev/null || echo "SCAN_FAILED: $img"
done
```

Save the combined JSON output to `.local/security/scans/YYYY-MM-DD-trivy-images.json`.

**If the Trivy safety check fails** (new advisory found, version mismatch, or image verification fails), skip Trivy scanning entirely and note in the report: "Trivy scanning skipped due to supply chain safety concern — manual review required." Use the alternative CVE sources in steps 3c and 3d instead.

#### 3e. Wazuh SCA (Security Configuration Assessment) Results

Query Wazuh's SCA database for CIS benchmark results:

```bash
# Discover the Wazuh manager container name
WAZUH_MGR=$(docker ps --format '{{.Names}}' | grep -i 'wazuh.*manager' | head -1)

# Get SCA scan results for the host agent
docker exec "$WAZUH_MGR" /var/ossec/bin/agent_control -l 2>/dev/null
# Use the agent ID found above (usually 001 for the host)

# Attempt SCA API query — this requires Wazuh API credentials.
# Discover the API user from docker-compose.yml environment variables
# (look for WAZUH_API_USERNAME or similar — do NOT read .env).
# If credentials cannot be discovered, skip SCA and note it in the report.
WAZUH_API_USER=$(docker inspect --format '{{range .Config.Env}}{{println .}}{{end}}' "$WAZUH_MGR" 2>/dev/null | grep -i 'API_USERNAME' | cut -d= -f2)
if [ -n "$WAZUH_API_USER" ]; then
  TOKEN=$(docker exec "$WAZUH_MGR" curl -sk -X POST \
    'https://localhost:55000/security/user/authenticate' \
    -H 'Content-Type: application/json' \
    -d '{"raw": true}' \
    -u "${WAZUH_API_USER}:" 2>/dev/null)
  docker exec "$WAZUH_MGR" curl -sk -X GET \
    "https://localhost:55000/sca/AGENT_ID?pretty=true" \
    -H "Authorization: Bearer $TOKEN" 2>/dev/null | head -100
  docker exec "$WAZUH_MGR" curl -sk -X GET \
    "https://localhost:55000/sca/AGENT_ID/checks?status=failed&pretty=true" \
    -H "Authorization: Bearer $TOKEN" 2>/dev/null | head -200
fi
```

Note: The Wazuh API credentials may be in `.env` which you must NOT read. If the API user cannot be discovered from the container environment and API calls fail due to authentication, skip SCA and note it in the report. The user can provide credentials verbally if needed.

#### Incident Escalation

If any finding is classified as **True Positive — Critical**, immediately:

1. **Containment first**: Propose commands to isolate the threat (stop container, block IP, disable account)
2. **Evidence preservation**: Before any destructive action, capture:
   ```bash
   # Snapshot container state before stopping
   docker inspect CONTAINER > /tmp/evidence-CONTAINER-$(date +%s).json
   docker logs CONTAINER > /tmp/evidence-CONTAINER-logs-$(date +%s).txt
   # Capture network connections
   ss -tulnp > /tmp/evidence-network-$(date +%s).txt
   # Capture process tree
   ps auxf > /tmp/evidence-procs-$(date +%s).txt
   ```
3. **Push notification**: If a push notification service (e.g., Gotify, ntfy) is available, send an immediate alert:
   ```bash
   # Discover push notification container and credentials from docker-compose.yml
   # Look for gotify, ntfy, or similar in the compose file — do NOT read .env
   # Example for Gotify (adapt based on what is discovered):
   # docker exec CONTAINER wget -qO- \
   #   --post-data='{"title":"CRITICAL SECURITY ALERT","message":"[description]","priority":10}' \
   #   --header='Content-Type: application/json' \
   #   'http://GOTIFY_HOST:PORT/message?token=APP_TOKEN' 2>/dev/null
   ```
   Discover the container name, port, and app token from `docker-compose.yml` environment variables (do NOT read `.env`).
4. **Recovery guidance**: After containment, provide steps to restore the affected service securely

#### Wazuh Active Response Recommendations

For true positive findings that could recur, propose Wazuh active response rules that would automate containment:

```xml
<!-- Example: auto-block an IP after repeated authentication failures -->
<command>
  <name>firewall-drop</name>
  <executable>firewall-drop</executable>
  <timeout_allowed>yes</timeout_allowed>
</command>

<active-response>
  <command>firewall-drop</command>
  <location>local</location>
  <rules_id>RULE_ID</rules_id>
  <timeout>3600</timeout>
</active-response>
```

Only propose active responses when:
- The trigger condition is precise (low false positive risk)
- The response is reversible (timeout-based)
- The blast radius is limited (blocking one IP, not disabling a service)

### Phase 5 — Supply Chain Assessment

As a final check, assess the freshness and trustworthiness of container images:

```bash
# Check image creation dates and tags
for cid in $(docker ps -q); do
  name=$(docker inspect --format '{{.Name}}' "$cid" | tr -d '/')
  image=$(docker inspect --format '{{.Config.Image}}' "$cid")
  created=$(docker inspect --format '{{.Created}}' "$cid")
  echo "$name | $image | created=$created"
done

# Check for images using :latest tag (no version pinning)
docker ps --format '{{.Image}}' | grep ':latest'

# Check for images older than 90 days
docker images --format '{{.Repository}}:{{.Tag}} {{.CreatedSince}}' | grep -E 'months|years'
```

Flag:
- Images using `:latest` tag (no reproducible builds)
- Images not updated in >90 days (may have unpatched vulnerabilities)
- Images from unverified registries

## Important Reminders

- **Local-only data**: Investigation reports (`.local/security/investigations/`), scan results (`.local/security/scans/`), and the false positive catalog (`.local/security/false-positives.yml`) must NEVER be committed to git. Always verify `.local/` is in `.gitignore`.
- **Trivy via Docker**: Always run Trivy as a Docker container with a pinned, verified safe version (see Phase 3a safety protocol) — do not install on the host and NEVER use `:latest`.
- **CVE prioritization**: CISA KEV matches > CRITICAL CVSS > HIGH CVSS. A KEV-listed vulnerability is confirmed exploited in the wild and takes absolute priority.
- **Compensating controls change risk ratings**: A CRITICAL CVE behind a zero-trust proxy with no direct network exposure is less urgent than a HIGH CVE on a service bound to `0.0.0.0`. Always factor in the defense-in-depth layers.
- **Trend matters**: A rule firing once is noise. The same rule firing with increasing frequency over weeks is a signal. Check historical data when available.
- **Active responses must be safe**: Only propose automated responses that are reversible, time-bounded, and precise. A bad active response rule is worse than no automation.
